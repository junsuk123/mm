# 상업화 및 수익 모델 가이드

이 데이터의 사업 가치는 “개인정보 판매”가 아니라, 동의·계약·품질 요건을 갖춘 집계 인사이트와 분석 서비스에서 만들어야 합니다.

## 가능한 수익 모델

### 1. 정기 상권 인사이트 구독

지역·끼니·음식군별 선호와 추천 결과를 월간 또는 분기 리포트로 제공합니다.

수익 구조 예:

- 상권·기간별 정액 구독
- 지역 수 또는 응답 규모에 따른 요금제
- 대시보드 계정과 CSV/API 제공을 묶은 기업 요금제

### 2. 맞춤형 분석 프로젝트

플랫폼이나 프랜차이즈가 지정한 상권·음식군에 맞춰 설문과 분석을 설계합니다.

수익 구조 예:

- 프로젝트 착수금
- 모집 응답 수 기반 비용
- 분석 보고서와 발표·컨설팅 비용

### 3. 추천 엔진·API 라이선스

데이터 원본 대신 그룹화, 최근 음식 회피, 제외 피드백, 거리 필터를 포함한 추천 로직을 제공합니다.

수익 구조 예:

- 월 API 호출량 기반 과금
- 기업 내부 설치형 라이선스
- 추천 정확도·운영 SLA가 포함된 유지보수 계약

### 4. 데이터 수집 SaaS

대학, 기업 구내식당, 행사, 단체 주문 운영자에게 QR 설문과 그룹 주문 기능을 제공합니다.

수익 구조 예:

- 세션 또는 참가자 수 기반 과금
- 관리자 대시보드 구독
- 브랜드 커스텀과 전용 도메인 비용

### 5. 데이터 가공·품질 서비스

플랫폼이 보유한 내부 데이터와 결합하기 전 표준화, 분류, 익명화, 리포트 제작 서비스를 제공합니다.

수익 구조 예:

- 데이터 가공량 기반 비용
- 품질 검수와 정기 업데이트 계약
- 데이터 가치평가·PoC 지원 비용

## 기업별 제안 방향 예시

- 배달 플랫폼: 그룹 주문, 끼니별 메뉴 수요, 교차 주문 카테고리
- 지도·검색 플랫폼: 그룹 외식 장소, 이동 허용 범위, 추천 제외 피드백
- 커머스 플랫폼: 간편식·밀키트 수요, 지역·시간대별 상품 구성

이는 제안 방향일 뿐 해당 기업의 구매 의사나 제휴를 의미하지 않습니다. 기업명과 상표는 제안서에서 사실관계와 계약 상태를 명확히 표시해야 합니다.

## 수익화 전 최소 기준

- 상업적 제3자 제공 목적을 반영한 수집 고지·동의 또는 다른 적법 근거
- 데모 데이터를 제외한 충분한 실제 표본
- 상권·기간·모집 방식 메타데이터
- 재식별 위험과 소수 셀 억제 기준
- 데이터 품질 지표와 오류 정정 절차
- 표준계약서 기반 사용권·보유기간·재제공 조건
- 수신 기업의 PoC 검수와 가격·업데이트 주기 합의

## 과장하면 안 되는 표현

- “대표성 있는 전국 소비자 데이터”
- “완전 익명이라 법적 검토 불필요”
- “추천 점수가 실제 구매 확률”
- “특정 플랫폼과 제휴된 데이터”

현재 데이터는 소규모 선호 조사와 추천 결과이므로, 초기에는 데이터 판매보다 PoC·분석 리포트·수집 SaaS·추천 API 형태가 현실적인 수익 경로입니다.
